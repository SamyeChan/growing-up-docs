## Day 01

目标：理解禅道管理流程的模型以及实际操作

### 禅道工具研发的背景、解决项目遇到的以下痛点

- 需求频繁变更
- 多次回归测试，Bug不能收敛
- 交付周期长
- 团队很努力，但无效率
- 承诺期限，但出现延期
- 人员流动大
- 交付不被市场/客户接受

### 禅道软件概述

- 禅道软件的特点
  1. 全生命周期管理
  2. 支持敏捷项目管理
  3. 易于扩展
- 禅道软件和同类软件比较
- 不同角色的操作简介
  1. 管理者
  2. 产品经理
  3. 项目经理
  4. 开发人员
  5. 测试人员
  6. 运维和运营人员

### 针对产品开发，禅道的流程管理模型 —— Scrum

- 典型 Scrum 管理框架
- Scrum 团队
  1. 跨职能团队
  2. 团队的角色协作方式
- 如何保证交付物和客户需求相符
  1. 建立需求列表（Product Backlog）
  2. 迭代增量交付
  3. 持续优先级管理
  4. 迭代验收，得到客户反馈
- 如何提升交付物的质量和效率
  1. 好的计划会议是交付成功的基石

    - 燃尽图，发现迭代的风险

    - 结对编程
    - 持续集成

### 禅道软件基于敏捷项目管理如何管理流程

总体思路：风险前移

#### 第一层改进——建立快速交付的流程

1. 需求池与短迭代
2. 时间盒
3. 透明管理
4. 自动化测试
5. 持续集成与重构
6. 客户参与

#### 第二层改进——优化流程中的需求管理

1. 需求管理
  - 建立需求列表
  - 迭代增量交付
  - 持续优先级管理
  - 需求变更管理
2. 如何判断需求的优先级
  - Kano 模型
  - 价值 vs 成本
  - 基于成本的需求分析
  - 高成本的解决方案
  - 高成本是风险的前兆
3. 快速得到客户的反馈

#### 第三层改进——优化流程中的项目管理

1. 优秀的项目团队
2. 透明化
3. 降低交付延期的风险——提升效率
4. 降低交付延期的风险——消除技术债务
5. 降低交付延期的风险——消除学习债务

【研发中的痛点在哪儿？】

#### 第四层改进——优化流程中的质量管理

1. 质量是制造出来，不是检测出来
2. 降低交付的失败风险
3. 减少需求缺陷风险
4. 减少构建缺陷风险
5. 高质量测试

#### 第五层改进——产品开发前的准备

1. Just Enough（正好够用）原则
2. 系统级需求
3. 架构级需求拆分
4. 开发级需求细化
5. 初始化架构设计
6. 增量迭代交付的计划制定

#### 第六层改进——产品上线后的及时响应

1. 迭代的响应太慢
2. 工作流模式（基于 Kanban）

#### 第七层改进——如何推广到更大的团队

1. 大型项目的团队组织架构
2. 大型项目的敏捷项目管理流程
